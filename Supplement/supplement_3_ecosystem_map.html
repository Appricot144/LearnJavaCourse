🔍 各層の詳細解説
🧑‍💻 あなたのアプリケーション
  • 本コースで書いたすべてのコードはここに該当します
  • 下の全ての層の機能を import して利用します
  • パッケージ設計（第14章）でフォルダ構造を整理します
📦 サードパーティライブラリ
  • JDK には含まれず、Maven / Gradle で追加するライブラリ群
  • Maven Central（世界最大のJavaリポジトリ）から自動ダウンロード
  • Spring Boot・Hibernate・JUnit など業界の事実上の標準が多い
  • Jakarta EE の仕様を実装しているものも多い（Hibernate は JPA の実装）
🏛️ Jakarta EE（旧 Java EE）
  • 大規模エンタープライズシステム向けの標準仕様群
  • 仕様（インターフェース定義）のみで、実装は各ベンダーが提供
  • JPA という仕様を Hibernate が実装、という関係が典型例
  • WildFly・Payara などのアプリサーバ上で動作する
  • 「Java EE」は Oracle → Eclipse Foundation に移管され「Jakarta EE」に改名
🪶 Apache Commons
  • Apache Software Foundation が管理するオープンソース汎用ライブラリ集
  • Java の標準ライブラリを補完する便利な機能を提供
  • commons-lang3：StringUtils、NumberUtils などの文字列・数値ユーティリティ
  • commons-io：ファイル操作を簡便にするクラス群
  • commons-collections4：Bag、MultiMap などの拡張コレクション
  • Spring Boot でも内部的に多く使われている
📚 Java 標準ライブラリ（JCL）
  • JDK に同梱。import するだけで追加設定なしに使えるクラス群
  • java.lang のみ自動 import。他は明示的に import が必要
  • 本コースでは java.util・java.util.stream・java.time などを多用
  • Java のバージョンアップごとに新しいクラス・メソッドが追加される
🔧 JDK（Java Development Kit）
  • 開発に必要なツール一式をまとめたパッケージ
  • javac（コンパイラ）、java（実行）、jar（パッケージング）を含む
  • Oracle JDK・Eclipse Temurin・Amazon Corretto などの「ディストリビューション」がある
  • すべて OpenJDK をベースにしており、動作は基本的に同じ
  • Java 11 以降は JRE（実行環境のみ）と JDK の区別がなくなった
⚙️ JVM（Java Virtual Machine）
  • バイトコードを実際の CPU・OS 命令に変換して実行する仮想マシン
  • JIT コンパイラがホットスポットを最適化し実行速度を向上させる
  • GC が不要なオブジェクトを自動解放（Stop-The-World が発生）
  • HotSpot（Oracle）・OpenJ9（IBM）・GraalVM などの実装がある
  • Kotlin・Scala・Groovy など他言語も JVM 上で動く
🔨 ビルドツール（Maven / Gradle）
  • すべての層をまたいで機能する「配管役」
  • サードパーティライブラリを Maven Central から自動取得
  • javac を呼び出してコンパイル、java でテスト実行を自動化
  • 最終的に .jar や .war にパッケージング
  • Maven は pom.xml、Gradle は build.gradle(.kts) で設定

💡 この図の読み方まとめ
  • 上の層ほどあなたのコードに近い。第1〜14章で書いたコードは一番上の層に位置します。
  • JDK（標準ライブラリ）は無条件に使える土台。import するだけで利用可能です。
  • Jakarta EE は「仕様書」、フレームワークはその「実装」。JPA を Hibernate が実装するように、仕様と実装は分離されています。
  • Apache Commons はどの層からでも使える補助ライブラリ。標準ライブラリを補完する位置づけです。
  • Maven / Gradle は「縦の配管役」。層をまたいでライブラリの取得・ビルド・テストを自動化します。
  • JVM はすべての土台。どのライブラリも最終的に JVM の上で動いています。
