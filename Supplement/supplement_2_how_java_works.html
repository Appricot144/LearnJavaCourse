🤔 そもそも「プログラムが動く」とはどういうことか

コンピュータの CPU が理解できるのは 0と1の機械語（マシンコード） だけです。
私たちが書く Java のコードは人間が読みやすい形で書かれており、そのままでは CPU は実行できません。

Java がどのように CPU で動くのかを理解するために、まず料理のレシピに例えて考えてみましょう。

フェーズ1：コンパイル（javac）

javac Main.java を実行すると、Javaコンパイラが以下を行います。

  • 文法チェック：括弧の対応・型の不一致・未定義の変数などのエラーをここで検出する
  • バイトコードへの変換：人間が読めるソースコードを「バイトコード」という中間形式に変換する
  • .class ファイルの生成：Main.class というファイルが生成される
ソースコード（人間が読める）→ バイトコード（機械寄りの中間コード） 
// Java ソースコード（Main.java）
System.out.println("Hello");

// ↓ javac でコンパイル後のバイトコード（javap -c Main で確認できる）
0: getstatic     #7   // Field java/lang/System.out:Ljava/io/PrintStream;
3: ldc           #13  // String Hello
5: invokevirtual #15  // Method println:(Ljava/lang/String;)V
8: return

// バイトコードは「JVMへの命令書」。OSや CPU の種類には依存しない

フェーズ2：実行（java / JVM）

java Main を実行すると、JVM（Java Virtual Machine）が Main.class を読み込んで実行します。

🏗️ JVM（Java Virtual Machine）の内部構造

JVM は「仮想的なコンピュータ」です。バイトコードを受け取り、実際に動いている OS・CPU の命令に変換して実行します。

▶ JVM の主要コンポーネント
JVM（Java Virtual Machine）
📥 クラスローダー（Class Loader）
.class ファイルをメモリに読み込む。必要なクラスを動的にロードする仕組み
🔍 バイトコードベリファイア
読み込んだバイトコードが安全かどうかを検証する。不正な操作を防ぐセキュリティ機構
📚 ランタイムデータ領域
ヒープ（オブジェクト）・スタック（メソッド呼び出し）・メソッド領域（クラス情報）など
🗑️ ガベージコレクター（GC）
不要になったオブジェクトを自動的にメモリから解放する。C言語のような手動メモリ管理は不要
▶️ 実行エンジン（Execution Engine）
バイトコードを実際に実行する核心部分。インタープリタと JIT コンパイラを含む
⚡ JIT コンパイラ（Just-In-Time Compiler）
頻繁に実行されるバイトコードをネイティブコード（機械語）に変換してキャッシュする。これにより実行が高速化する
🌏 なぜどのOSでも動くのか — Write Once, Run Anywhere

Java の有名なキャッチフレーズ「Write Once, Run Anywhere」（一度書けば、どこでも動く）の秘密はバイトコードにあります。

⚡ JIT コンパイラ — 実行しながら高速化する仕組み

JVM がバイトコードを実行する方法は2つあります。

🐢 インタープリタ方式（起動直後）
バイトコードを1行ずつ読んでその場で実行する。
起動は速いが、繰り返し実行される処理は毎回変換が走るため遅い。
🐇 JIT コンパイル方式（ホットスポット検出後）
頻繁に実行されるコード（ホットスポット）を検出し、ネイティブコードに変換してキャッシュする。
2回目以降は直接 CPU が実行するため大幅に高速化。
💡 JVM の実装「HotSpot」とは
Oracle（および OpenJDK）が開発した JVM の実装名が「HotSpot」です。
頻繁に実行されるコード（ホットスポット）を検出して JIT コンパイルする技術から命名されました。
java -version を実行したときに「HotSpot VM」と表示されるのがこれです。
🧹 メモリ管理とガベージコレクション（GC）

Java プログラムが実行されるとき、JVM はメモリをいくつかの領域に分けて管理しています。

▶ JVM のメモリ領域
🏔️ ヒープ（Heap）
new で生成したオブジェクトが置かれる場所。
GC が管理し、参照されなくなったオブジェクトを自動解放する。
new Dog(), new ArrayList<>(), …
📚 スタック（Stack）
メソッド呼び出しのたびに「フレーム」が積まれる。
ローカル変数・引数・戻り先アドレスが保存される。メソッド終了で自動解放。
int x = 5; の x など
📖 メソッド領域
クラスの定義情報（フィールド名・メソッドのバイトコードなど）が格納される。
static フィールドもここに置かれる。
static int count; など

ガベージコレクション（GC）の仕組み

Java ではどこからも参照されなくなったオブジェクトを GC が自動的にメモリから解放します。
C言語のように手動で free() を呼ぶ必要がありません。

⚠️ GC のデメリット — STW（Stop The World）
GC が実行される瞬間、一時的にアプリケーションの処理が止まります（Stop The World）。
これが「Java は遅い」と言われる原因の一つです。ただし現代の JVM（G1 GC・ZGC など）は
この停止時間を数ミリ秒以下に抑えるよう最適化されており、多くのケースで問題になりません。
🧵 スレッド — 並列処理の仕組み

プログラムは通常1本の「実行の流れ（スレッド）」で動いています。
Java の JVM は複数のスレッドを同時に動かす（マルチスレッド）ことができます。

ヒープは共有されるため複数スレッドが同じオブジェクトを触れる → 同期処理（synchronized）が重要になる

マルチスレッドの簡単な例
// 新しいスレッドを作って並行処理
Thread t = new Thread(() -> {
    System.out.println("別スレッドで実行！");
});
t.start();  // 新しいスレッドが起動する
System.out.println("メインスレッドで実行！");

// どちらが先に出力されるかは実行タイミング次第（非決定的）
